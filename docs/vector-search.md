# Поиск релевантных постов — исследование и итоговая архитектура

## Задача

Для каждой темы контент-плана найти топ-15 постов из корпуса (~11k постов, 9 каналов), которые могут дать идеи, факты, истории или формулировки для написания нового поста.

Метрика: **must@15** (обязательные посты попали в топ-15) и **relevant@15** (сколько из всех релевантных постов попало).

---

## Методы поиска — эволюция

### 1. BM25 (базовый)

TF-IDF с нормализацией длины документа. Используется `wink-bm25-text-search` с русским стеммером (`@nlpjs/lang-ru`).

```javascript
search(query, topK)
```

**Проблема**: "яхтенные путешествия" не находит посты где написано "море — это кайф и свобода". BM25 работает по точным словам.

---

### 2. Query Expansion (BM25 + синонимы)

Перед BM25 просим GPT-4o-mini дать 5-7 синонимов запроса, добавляем их к исходному тексту запроса.

```javascript
expandQuery("яхтенные путешествия")
// → ["яхтенные путешествия", "морской туризм", "парусный отдых", "круиз на яхте", ...]
```

Небольшое улучшение для коротких запросов, но не решает проблему фреймового разрыва.

---

### 3. Векторный поиск (dense retrieval)

Каждый пост эмбеддируется при индексировании. При поиске — эмбеддим запрос и считаем косинусное сходство.

Модель: `text-embedding-3-small` (OpenAI), 1536 измерений, $0.02 за 1M токенов.

**Проблема**: прямой эмбеддинг запроса "Как выйти замуж на яхте" содержит паттерн "Как X на яхте" и ближе к другим "как X на яхте"-постам, чем к постам про свадьбы.

---

### 4. Query Rewriting (перед эмбеддингом)

Перед генерацией эмбеддинга переписываем запрос в ключевые понятия — существительные и прилагательные без вопросительной структуры.

```javascript
rewriteQueryForEmbedding("Как выйти замуж на яхте")
// → "свадьба на яхте, замужество, венчание, предложение руки и сердца"
```

Этот вектор семантически точнее попадает в посты о свадьбах.

---

### 5. HyDE (Hypothetical Document Embeddings)

Вместо эмбеддинга запроса — просим GPT написать гипотетический пост на эту тему, эмбеддим его.

```
Запрос: "Как выйти замуж на яхте"
HyDE: "Романтическая свадьба прямо в море — это реально! Мы организовали
       венчание на борту нашей яхты..."
```

Гипотетический пост использует те же слова и фреймы что и реальные посты → вектор ближе к релевантным документам. Особенно помогает при большом фреймовом разрыве (спрашивающий vs организатор).

Стоимость: +1 вызов GPT-4o-mini (~$0.0001) + +1 эмбеддинг на запрос.

---

### 6. Hybrid Search (BM25 + Vector)

```
score = 0.3 * bm25_normalized + 0.7 * cosine_similarity
```

BM25 хорош для точных терминов, вектор — для семантики. `alpha=0.3` эмпирически лучший на нашем корпусе.

---

### 7. RRF (Reciprocal Rank Fusion)

Объединяет несколько ранкингов без нормализации скоров:

```
rrf_score = Σ 1/(k + rank_i)    k=60 (стандартный параметр)
```

**Преимущество** над взвешенным суммированием: разные сигналы имеют разные масштабы скоров. RRF работает с позициями — они всегда сопоставимы.

```javascript
hybridSearchRRF(query)  // BM25 + rewritten-vector + HyDE → RRF
```

---

### 8. Multi-Query BM25 (multiQueryRRF)

Проблема BM25: один запрос покрывает только один словарь. Генерируем 4 парафразы в стиле русских соцсетей, прогоняем BM25 по каждой, объединяем 5 ранкингов через RRF.

```
Запрос: "яхтенные путешествия — лучший отдых"
Парафраза 1: "почему отдыхать на яхте кайф"
Парафраза 2: "яхтинг как идеальный отпуск"
Парафраза 3: "почему стоит выбрать морской круиз"
Парафраза 4: "лучший отдых — море и паруса"
```

Каждая парафраза ловит другое подмножество постов с разными словами.

---

### 9. mqHybridRRF — основной метод до re-ранкинга

Объединяет все сигналы в один RRF:

| Сигнал | Откуда |
|---|---|
| BM25 (оригинал) | Точные термины из запроса |
| BM25 (парафраза 1-4) | Разные варианты словаря |
| Vector (rewritten) | Семантика без вопросит. структуры |
| HyDE | Устраняет фреймовый разрыв |

**Итого: 7 ранкингов в RRF.**

Реализация: генерируем парафразы, rewrite и HyDE параллельно, затем параллельно генерируем два эмбеддинга.

---

### 10. searchWithRerank — финальный метод

```
mqHybridRRF(top-100 кандидатов)
  → LLM listwise re-rank (gpt-4o, выбирает top-30 из 100)
  → RRF(re-rank top-30, mqHybrid top-15)
  → топ-15
```

**LLM re-ranking**: отправляем 100 кандидатов с сниппетами в gpt-4o, просим выбрать N наиболее релевантных по номерам. Listwise — один запрос на все кандидаты (vs pointwise — один запрос на кандидата, упирается в лимиты RPM).

**Ensemble через RRF**: Re-ranker может промахнуться (позиционный bias, длинные списки). Объединяем re-rank топ-30 и mqHybrid топ-15 через RRF — берём лучшее из обоих.

**Сниппеты для длинных постов**: тема может раскрываться не только в начале. "Бутерброд":
```javascript
text.length > 600
  ? text.slice(0, 350) + ' … ' + text.slice(-150)
  : text.slice(0, 500)
```

## Чанкинг

Один эмбеддинг на весь пост усредняет все темы. Чанкинг разбивает пост на фрагменты, каждый индексируется отдельно. При поиске — находим лучший чанк и возвращаем родительский пост.

**Стратегия** (`search/chunking.js`):
1. Делим по абзацам (`\n`)
2. Абзац > 120 слов → делим по предложениям
3. Чанки < 30 символов отбрасываем

**Почему не помогло на нашем корпусе**: ~80% постов короткие (Instagram/Telegram, ~60 слов). Один пост = один чанк. Chunking полезен для длинных документов с несколькими темами.

## Результаты тестирования

**Тестовый набор**: фикстуры в `tests/fixtures/` — по 50 постов на канал, все релевантные/обязательные посты гарантированно включены (REQUIRED_IDS в `scripts/create-fixtures.js`).

**Метрика**: топ-15 из 450 постов (9 каналов × 50 постов).

**Потолок recall**: 96/105 — не все релевантные посты физически есть в фикстурах.

### Анализ провалов

**T2 "Как проходит один день на яхте"** — стабильно плохо. Релевантные посты описывают конкретные события ("поехали за провизией", "утром — подъём паруса"), а не "типичный день". Семантический gap слишком широк даже для HyDE.

**T10 "История: как мы сели на мель"** — re-ranker читает пост `LyubimovaEvgeniya_2021` ("мель" упоминается не в начале текста, а дальше) и не всегда подхватывает его. Сниппет-бутерброд помогает, но не стабильно.

**T1 "Яхтенные путешествия — лучший отдых"** — посты используют "кайф", "свобода", "инвестиция в себя" вместо "отдых". BM25 провал, vector/HyDE частично помогают. Re-ranker справляется.

## Стоимость запроса

### Текущий корпус (~11k постов, `searchWithRerank`)

| Операция | Модель | Стоимость |
|---|---|---|
| Парафразы (4 шт.) | gpt-4o-mini | ~$0.00003 |
| Query rewrite | gpt-4o-mini | ~$0.00001 |
| HyDE | gpt-4o-mini | ~$0.00003 |
| 3 эмбеддинга (rewrite + HyDE + запрос) | text-embedding-3-small | ~$0.000006 |
| LLM re-rank (100 кандидатов × ~150 токенов) | gpt-4o | ~$0.002 |
| **Итого** | | **~$0.002 за запрос** |

**Примечание**: BM25 и cosine similarity — бесплатны (CPU), занимают <50мс.

### Масштаб корпуса

Re-ranking работает только с топ-100 кандидатами, поэтому его стоимость **не зависит от размера корпуса**. Растут только косинусные вычисления.

| Корпус | Векторная часть (RAM) | Cosine similarity | Подход |
|---|---|---|---|
| 11k постов | ~67MB | ~10мс | В памяти ✅ |
| 100k постов | ~600MB | ~100мс | В памяти (возможно) |
| 1M постов | ~6GB | ~1сек | Нужна векторная БД |

**При 100k постах**: те же $0.002 за запрос. Нужно смотреть на RAM (~600MB) и время cosine (~100мс). Приемлемо.

**При 1M постах**: та же стоимость GPT, но cosine similarity по 1M векторов в памяти (~1сек) и 6GB RAM. Нужна векторная БД — Qdrant или Chroma, которая индексирует векторы и делает approximate nearest neighbors за <50мс.

### Стоимость индексирования

| Операция | Стоимость |
|---|---|
| Эмбеддинги 11k постов (full-post) | ~$0.018 |
| Эмбеддинги 11k постов (chunk) | ~$0.022 |
| Разница | +$0.004 |

Индексирование — разовая операция. При 100k постов: ~$0.16 (full-post). При 1M: ~$1.6.

## Длинные посты: как улучшить поиск без платных вызовов

**Проблема**: длинный пост может упоминать тему только в середине или конце. Эмбеддинг всего поста "разбавляется" другими темами. BM25 с тфидф работает по всему тексту, поэтому не затронут.

### Что уже сделано (для re-ranker)

Сниппет-бутерброд в `llmRerank` — начало + конец:
```javascript
text.length > 600
  ? text.slice(0, 350) + ' … ' + text.slice(-150)
  : text.slice(0, 500)
```

### Бесплатные подходы для эмбеддингов длинных постов

**Скользящее окно** — эмбеддируем пост кусками по 200 слов с перекрытием 50, берём максимальное сходство. Чанкинг уже реализован в `search/chunking.js`.

**Extractive summary (без API)** — TF-IDF по предложениям: каждое предложение оцениваем по сумме IDF весов слов, берём топ-3. Библиотека `node-summarizer` или `natural`. Бесплатно, работает локально.

```javascript
// Псевдокод extractive summary
function extractTopSentences(text, n = 3) {
  const sentences = text.split(/[.!?]+/).filter(s => s.length > 20);
  const wordFreq = buildTfIdf(sentences);
  return sentences
    .map(s => ({ s, score: scoreByTfIdf(s, wordFreq) }))
    .sort((a, b) => b.score - a.score)
    .slice(0, n)
    .map(x => x.s)
    .join(' ');
}
```

Для эмбеддинга индексировать `extractTopSentences(text)` вместо полного текста. Применимо только для длинных постов (>300 слов, ~5% корпуса).

**Вывод**: наш корпус в основном короткий (~60 слов, 80% постов = 1 чанк), поэтому эффект будет заметен только для каналов с длинными постами (например `LyubimovaEvgeniya`).

## Кэширование GPT-вызовов

Все дорогие GPT-вызовы кэшируются в двух уровнях:

1. **In-memory** (Map): повторный вызов в том же процессе → мгновенно
2. **Disk** (`.cache/queries/*.json`): между запусками → детерминированные тесты, экономия на API

Ключ кэша: тип + base64url(query), первые 64 символа. При изменении запроса — новый файл.

```
.cache/queries/
  para_<hash>.json      # 4 паrafразы
  hyde_<hash>.json      # гипотетический документ
  rewrite_<hash>.json   # переписанный запрос
```

## Открытые вопросы и возможные улучшения

- **T2 "Один день на яхте"** — need ground truth расширить или добавить посты про распорядок дня
- **BM25 стеммер** — `StemmerRu` из `@nlpjs/lang-ru`. Работает нормально для большинства форм. Кейс "мели" → "мел" вместо "мель" — известный edge case.
- **Positional bias LLM re-ranker** — gpt-4o тяготеет к постам в начале списка. Тестировалось перемешивание — ухудшило результат (GPT хуже работает с длинными перемешанными списками). Оставлено без изменений.
- **Два-pass batching** — разбивка 100 кандидатов на 2×50 для уменьшения контекста. Попробовали — must posts не выживают первый pass. Отказались.
- **Pointwise re-ranking** — оценка каждого кандидата от 1 до 10 в параллельных запросах. Упёрлись в лимит 500 RPM. Откатились на listwise.
- **Масштабирование** — при 100k+ постов нужна векторная БД (Qdrant, Chroma). Стоимость GPT-части не изменится.
