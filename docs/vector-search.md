# Векторный поиск — реализация, чанкинг и результаты тестирования

## Идея

Текущий BM25-поиск работает по точным словам: "акула" находит посты где написано "акула", но не находит посты про "нападение в море" или "опасный обитатель".

Векторный поиск превращает текст в числовой вектор (эмбеддинг) так, что близкие по смыслу тексты дают близкие векторы. Запрос "как провести день на яхте" найдёт пост "утро начинается с подъёма паруса" даже без совпадения слов.

---

## Архитектура

### Хранение эмбеддингов

```
data/
  posts/
    seapinta.json              # посты
  embeddings/
    seapinta.json              # [{ id, vector }] — один вектор на пост
  chunk-embeddings/
    seapinta.json              # [{ postId, chunkIdx, vector }] — вектор на каждый чанк
```

### Модель

`text-embedding-3-small` (OpenAI)
- 1536 измерений
- $0.02 за 1M токенов

### Режимы поиска

| Функция | Описание |
|---|---|
| `search()` | BM25 + расширение синонимами (OpenAI) |
| `vectorSearch()` | Косинусное сходство эмбеддинга **переписанного** запроса с эмбеддингами постов |
| `hybridSearch()` | `alpha * bm25 + (1-alpha) * vector`, alpha=0.3 |
| `vectorSearchChunked()` | Как vectorSearch, но сравнивает с чанками постов |
| `hybridSearchChunked()` | Как hybridSearch, но vector-часть — по чанкам |

### Гибридная формула

```
finalScore = alpha * bm25Score + (1 - alpha) * vectorScore
```

`alpha = 0.3` — BM25 хорошо находит точные имена и термины, vector — смысл.

## Чанкинг

### Зачем

Один эмбеддинг на весь пост усредняет все темы поста. Если пост о регате содержит одну фразу "предложение руки и сердца", этот фрагмент тонет в среднем векторе.

Чанкинг разбивает пост на фрагменты, каждый индексируется отдельно. При поиске находим наиболее похожий чанк и возвращаем родительский пост.

### Стратегия (`search/chunking.js`)

1. Делим текст по абзацам (`\n`)
2. Если абзац > 120 слов — делим дальше по предложениям
3. Чанки < 30 символов отбрасываем

### Результаты на нашем корпусе

Средний пост — **60 слов** (большинство Instagram/Telegram постов короткие).
- Постов с 1 чанком: ~80%
- Постов с 2–3 чанками: ~15%
- Постов с 4+ чанками: ~5%

**Multiplier по токенам: ~1.3x**

### Стоимость (на 11k постов)

| Вариант | Токенов | Стоимость |
|---|---|---|
| Full-post embeddings | ~880k | ~$0.018 |
| Chunk embeddings | ~1.1M | ~$0.022 |
| Разница | +220k | +$0.004 |

Практически negligible. Chunk-эмбеддинги имеет смысл генерить сразу.

## Методы поиска

| Функция | Описание |
|---|---|
| `search()` | BM25 + расширение синонимами (OpenAI) |
| `vectorSearch()` | Эмбеддинг **переписанного** запроса (noun phrases) → cosine similarity |
| `hybridSearch()` | `0.3*bm25 + 0.7*vector` |
| `vectorSearchChunked()` | Как vectorSearch, но по чанкам постов |
| `hybridSearchChunked()` | Hybrid с chunk-векторами |
| `vectorSearchHyDE()` | Векторный поиск по гипотетическому документу |
| `hybridSearchHyDE()` | BM25 + HyDE-вектор |
| `hybridSearchFull()` | `0.2*bm25 + 0.4*vector + 0.4*hyde` |
| **`hybridSearchRRF()`** | **RRF по трём ранкингам: BM25 + rewritten-vector + HyDE** |

### Query Rewriting (перед эмбеддингом)

Проблема прямого эмбеддинга запроса "Как выйти замуж на яхте": вектор запроса содержит структурный паттерн "Как X на яхте" и ближе к постам "Как крепить гамак на яхте", чем к постам о свадьбе.

Решение: перед генерацией эмбеддинга переписываем запрос в ключевые понятия — существительные и прилагательные без вопросительной структуры.

Пример: "Как выйти замуж на яхте" → "свадьба на яхте, замужество, венчание, предложение руки и сердца" → этот вектор ближе к постам о свадьбах.

Применяется в: `vectorSearch`, `hybridSearch`, `vectorSearchChunked`, `hybridSearchChunked`, вектор-сигнал в `hybridSearchRRF`.

### HyDE (Hypothetical Document Embeddings)

Вместо эмбеддинга запроса — просим GPT написать гипотетический пост на эту тему, затем эмбеддим его. Гипотетический пост содержит те же слова и фреймы что и реальные посты → вектор ближе к релевантным документам.

Пример: запрос "Как выйти замуж на яхте" → GPT генерирует "Романтическая свадьба прямо в море — это реально! Мы организовали венчание на борту..." → этот вектор близок к постам про организацию свадеб на яхте.

Ограничение: +1 вызов GPT-4o-mini (~$0.0001) + +1 эмбеддинг на запрос. Кэшируется в памяти.

### RRF (Reciprocal Rank Fusion)

Объединяет три ранкинга без нормализации скоров:

```
rrf_score = 1/(60+rank_bm25) + 1/(60+rank_vector) + 1/(60+rank_hyde)
```

Преимущество над взвешенным суммированием скоров: разные методы имеют разные масштабы и распределения скоров — нормализация теряет информацию. RRF работает с позициями, которые всегда сопоставимы.

## Результаты тестирования Hit@5

Тестовый набор: 50 постов на канал (фикстуры в `tests/fixtures/`), 10 тем из контент-плана.

### Итоговый счёт

| Метод | Hit@5 из 10 |
|---|---|
| BM25 | 3–6 (нестабильно — зависит от random-филлера в фикстурах) |
| Vector (full-post) | 6 |
| Hybrid | 5–7 |
| HyDE | 4–5 |
| **RRF (BM25 + Vector + HyDE)** | **8** |

| Тема | BM25 | Vector | Hybrid | HyDE | **RRF** |
|---|---|---|---|---|---|
| Почему яхтенные путешествия лучший отдых | ✅5 | ❌ | ✅3 | ❌ | ✅3 |
| До конца предпродажи 6 дней | ✅1 | ✅1 | ❌ | ❌ | ✅3 |
| **Как проходит один день на яхте** | ❌ | ❌ | ❌ | ❌ | ❌ |
| Как выйти замуж на яхте | ❌ | ❌ | ❌ | ❌ | **✅4** |
| Как попадают на первую яхту | ❌ | ❌ | ✅5 | ❌ | ✅3 |
| 5 вещей, которые не ожидают | ❌ | ✅3 | ✅3 | ✅2 | ✅4 |
| Регата в Турции 5 лодок | ✅3 | ✅2 | ❌ | ✅2 | ✅2 |
| История: акула и камера | ✅4 | ✅2 | ✅3 | ✅2 | ✅4 |
| Самые красивые бухты Турции | ❌ | ✅2 | ✅1 | ✅1 | ✅1 |
| **История: как мы сели на мель** | ❌ | ✅3 | ❌ | ❌ | ❌ |

### Анализ провалов

**"Как выйти замуж на яхте"** — BM25, Vector, Hybrid не находят. **RRF находит (pos 4)** через HyDE-сигнал. HyDE генерирует гипотетический пост про свадьбу на яхте → его вектор оказывается близким к `seapinta_549` ("устраиваю свадьбы и венчания").

**"Как проходит один день на яхте"** — стабильно падает везде. Relevant посты (`meetingplace_news_367`, `meetingplace_news_32`, `regataveka_64`) описывают конкретные события, а не "типичный день". Семантический gap слишком широкий даже для HyDE. Нужно расширить ground truth или добавить больше постов про распорядок дня на яхте.

**"История: как мы сели на мель"** — только Vector находит (pos 3). В гибридах BM25-шум перебивает vector-сигнал. В RRF BM25 вытаскивает нерелевантные посты с "сели" в других контекстах.

**BM25 нестабилен** — результат меняется при каждом запуске из-за random-филлера в фикстурах. Постоянны только провалы для семантических запросов без ключевых слов (стемминг английский).

### Почему чанкинг не помог

Большинство постов (80%) короткие и уже помещаются в один чанк. Чанкинг полезен для длинных документов с несколькими темами. Для нашего Instagram/Telegram корпуса (~60 слов среднее) — эффект минимален.

## Известные ограничения

- **BM25** — `StemmerRu` (`@nlpjs/lang-ru`): русские падежные формы приводятся к основе ("мели" → "мел", "яхтой" → "яхт"). Русские стоп-слова фильтруются.
- **Dense retrieval** — один вектор усредняет всё содержимое поста. Решения: chunking (реализован), ColBERT (не реализован, сложнее).
- **Instagram CDN-ссылки** — URL картинок истекают через ~24-48ч. OCR backfill нужно запускать сразу после сбора.
- **11k постов в RAM** — 11k × 1536 float32 = ~67MB. Приемлемо. При 100k+ постов нужна векторная БД (Qdrant, Chroma).

```bash
# Рекомендуемый метод для продакшна
hybridSearchRRF(query, 10, { postsDir, embeddingsDir })
```

## Открытые вопросы

- **Когда обновлять эмбеддинги?** При перескрапинге или изменении `textClean`. Сейчас посты не редактируются после сохранения.
- **Память** — 11k × 1536 float32 ≈ 67MB. При 100k+ постах нужна векторная БД.
- **Скорость** — cosine similarity по 11k векторов ≈ 10мс на Node.js. Приемлемо.
- **Chunked vs Full-post** — при текущем корпусе разница минимальна. Можно включить chunked как основной режим без потери качества.
