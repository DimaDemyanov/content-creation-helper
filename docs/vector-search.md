# Векторный поиск — реализация, чанкинг и результаты тестирования

## Идея

Текущий BM25-поиск работает по точным словам: "акула" находит посты где написано "акула", но не находит посты про "нападение в море" или "опасный обитатель".

Векторный поиск превращает текст в числовой вектор (эмбеддинг) так, что близкие по смыслу тексты дают близкие векторы. Запрос "как провести день на яхте" найдёт пост "утро начинается с подъёма паруса" даже без совпадения слов.

---

## Архитектура

### Хранение эмбеддингов

```
data/
  posts/
    seapinta.json              # посты
  embeddings/
    seapinta.json              # [{ id, vector }] — один вектор на пост
  chunk-embeddings/
    seapinta.json              # [{ postId, chunkIdx, vector }] — вектор на каждый чанк
```

### Модель

`text-embedding-3-small` (OpenAI)
- 1536 измерений
- $0.02 за 1M токенов

### Три режима поиска

| Функция | Описание |
|---|---|
| `search()` | BM25 + расширение синонимами (OpenAI) |
| `vectorSearch()` | Косинусное сходство эмбеддинга запроса с эмбеддингами постов |
| `hybridSearch()` | `alpha * bm25 + (1-alpha) * vector`, alpha=0.3 |
| `vectorSearchChunked()` | Как vectorSearch, но сравнивает с чанками постов |
| `hybridSearchChunked()` | Как hybridSearch, но vector-часть — по чанкам |

### Гибридная формула

```
finalScore = alpha * bm25Score + (1 - alpha) * vectorScore
```

`alpha = 0.3` — BM25 хорошо находит точные имена и термины, vector — смысл.

## Чанкинг

### Зачем

Один эмбеддинг на весь пост усредняет все темы поста. Если пост о регате содержит одну фразу "предложение руки и сердца", этот фрагмент тонет в среднем векторе.

Чанкинг разбивает пост на фрагменты, каждый индексируется отдельно. При поиске находим наиболее похожий чанк и возвращаем родительский пост.

### Стратегия (`search/chunking.js`)

1. Делим текст по абзацам (`\n`)
2. Если абзац > 120 слов — делим дальше по предложениям
3. Чанки < 30 символов отбрасываем

### Результаты на нашем корпусе

Средний пост — **60 слов** (большинство Instagram/Telegram постов короткие).
- Постов с 1 чанком: ~80%
- Постов с 2–3 чанками: ~15%
- Постов с 4+ чанками: ~5%

**Multiplier по токенам: ~1.3x**

### Стоимость (на 11k постов)

| Вариант | Токенов | Стоимость |
|---|---|---|
| Full-post embeddings | ~880k | ~$0.018 |
| Chunk embeddings | ~1.1M | ~$0.022 |
| Разница | +220k | +$0.004 |

Практически negligible. Chunk-эмбеддинги имеет смысл генерить сразу.

### Анализ провалов

**"Как выйти замуж на яхте"** — падает везде. Причина: relevant посты (`seapinta_549` — "устраиваю свадьбы", `ig_clevel.yacht_DPvo6ACCtm7` — "предложение руки и сердца" как одна фраза в посте про регату) семантически далеки от запроса. Ground truth требует пересмотра.

**"Как проходит один день на яхте"** — relevant посты `meetingplace_news_367`, `meetingplace_news_32`, `regataveka_64` описывают конкретные события, а не "типичный день". Семантический gap слишком широкий. Возможное решение: добавить больше relevant постов в ground truth.

**BM25 нестабилен** — результат меняется при каждом запуске из-за random-филлера в фикстурах. Постоянны только провалы для семантических запросов без ключевых слов в тексте поста (стемминг английский, русские слова без лемматизации).

### Почему чанкинг не помог

Большинство постов (80%) короткие и уже помещаются в один чанк. Чанкинг полезен когда:
- Длинные посты с несколькими темами
- Релевантный фрагмент — одна-две фразы в длинном тексте

Для нашего корпуса Instagram/Telegram эффект минимален. Если бы посты были статьями по 1000+ слов — выигрыш был бы существенным.

## Известные ограничения

- **BM25** — Porter stemmer английский. "Шторма" и "шторм" — разные токены. Расширение синонимами через OpenAI частично компенсирует.
- **Dense retrieval** — один вектор усредняет всё содержимое поста. Решения: chunking (реализован), ColBERT (не реализован, сложнее).
- **Instagram CDN-ссылки** — URL картинок истекают через ~24-48ч. OCR backfill нужно запускать сразу после сбора.
- **11k постов в RAM** — 11k × 1536 float32 = ~67MB. Приемлемо. При 100k+ постов нужна векторная БД (Qdrant, Chroma).

## Команды

```bash
# Сгенерировать full-post эмбеддинги
node --env-file=.env scripts/backfill-embeddings.js

# Сгенерировать chunk-эмбеддинги
node --env-file=.env scripts/backfill-embeddings.js --chunked

# Один канал
node --env-file=.env scripts/backfill-embeddings.js --channel seapinta

# Для фикстур
node --env-file=.env scripts/backfill-embeddings.js --postsDir tests/fixtures/posts --embeddingsDir tests/fixtures/embeddings
node --env-file=.env scripts/backfill-embeddings.js --chunked --postsDir tests/fixtures/posts --chunkEmbeddingsDir tests/fixtures/chunk-embeddings

# Запустить интеграционные тесты
npm run test:integration
```

## Открытые вопросы

- **Когда обновлять эмбеддинги?** При перескрапинге или изменении `textClean`. Сейчас посты не редактируются после сохранения.
- **Память** — 11k × 1536 float32 ≈ 67MB. При 100k+ постах нужна векторная БД.
- **Скорость** — cosine similarity по 11k векторов ≈ 10мс на Node.js. Приемлемо.
- **Chunked vs Full-post** — при текущем корпусе разница минимальна. Можно включить chunked как основной режим без потери качества.
